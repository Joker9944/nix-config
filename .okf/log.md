# Update Log

## 2026-07-24

* **Fix**: Ported the greetd (regreet) Hyprland config from hyprlang to Lua. hyprlang is deprecated and current Hyprland parses `--config` files as Lua, so the greeter's `toHyprconf`-generated config threw Lua syntax errors (`syntax error near '-'` on `exec-once=`) and never came up — login was only possible from a text VT. The main session was already Lua (`configType = "lua"`); only the greeter lagged. home-manager 26.05 exposes **no reusable Hyprland Lua generator** (the `hl.config`/`hl.window_rule`/`hl.on` rendering is inline `let`-logic in its module), so `modules/nixos/extensions/regreet/hyprland.nix` now hand-rolls a small renderer over nixpkgs' `lib.generators.toLua`, plus an `execOnce` list option rendered into one `hl.on("hyprland.start", …)` hook. Mixin (`hosts/mixins/…/regreet.nix`) and host override (`hosts/HAL9000/default.nix`) ported to the Lua schema (`settings.config.{input,misc}`, `settings.window_rule`). Verified `Hyprland --config <generated.lua> --verify-config` → `config ok` and the graphical greeter returns.

## 2026-07-23

* **Tooling refactor**: The options-lookup engine is now a Nix factory (`mkOptionsTool`) that bakes a dataset's `options.json` into a small `writeShellApplication` binary — one per dataset (`hm-options`, `nixos-options`). This drops the old per-call `nix build` (and the `hm-doc` path-cache from `86859d8`): every query is just `jq` (~46 ms), and the binaries land on `PATH` via a new `devShells.default` + `.envrc` (`use flake`, `nix-direnv`-cached). Removed the `home-manager-docs-json` / `nixos-docs-json` **package outputs** — they existed only to be `nix build`-able; the JSON is now vendored into the tools. The NixOS JSON is a bare `eval-config` (`modules = [ ]`) → vanilla in-tree options only (cache-hits the Hydra `options` artifact; excludes this repo's mixins/external modules unless `documentation.nixos.includeAllModules = true`). The factory, JSON resolution, and engine (`files/nix-options.sh`) live under `pkgs/nix-options/`, whose `default.nix` returns `{ hm-options; nixos-options; }` directly. `flake.nix` stays lean — it only wires the dev shell to `self.packages.<system>.{hm,nixos}-options`. `pkgs/default.nix` now discovers in two passes: top-level `.nix` files → `callPackage` (get `flake`), and top-level subdirectories → `import path args` (get full args incl. `inputs`), each returning an attrset of packages merged into the set. So a file is one package and a subdir is a flake-aware package-group; `inputs` is threaded via `args` rather than `callPackage`. Captured as new concept [packages](/architecture/packages.md). Deleted `.claude/skills/home-manager-options/scripts/hm-doc`; the `home-manager-options` skill now calls the on-`PATH` `hm-options`. Updated [lookup-hm-option](/workflows/lookup-hm-option.md) and the `CLAUDE.md` companion-tooling bullet. **Next step**: a dedicated `nixos-options` skill (SKILL.md + NixOS trap table + evals) consuming the already-built `nixos-options` binary. `nix-darwin` would slot in as one more `mkOptionsTool` call when it becomes a flake input.
